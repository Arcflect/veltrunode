# 貢献ガイド (Contributing Guide)

Veltrunode プロジェクトへの貢献をご検討いただきありがとうございます！バグ報告、機能提案、ドキュメントの改善、プルリクエストなど、あらゆる形での貢献を歓迎します。

## 開発の原則（重要）

クリーンルーム開発と知的財産の保護のため、以下の原則を厳守してください。

- **他の類似ツールやフレームワーク等のコードやドキュメントを、コピーまたは機械的に翻案・翻訳したものを提出しないでください。**
- すべての機能提案や実装は、AWSの公開仕様、独立したユーザー要件、および独自設計に基づいている必要があります。

## 機能提案のプロセス

大きな変更や新しい機能を追加する場合は、コードを書く前にまず **Issue** を起票してください。提案には以下の情報を含める必要があります。

1. **ユーザーの課題**: どのような運用上の課題や問題点を解決するのか。
2. **AWSの公開仕様**: 対象となるAWSサービス（Lambda、Scheduler、EFSなど）のどの仕様に基づいているか。
3. **独自の設計**: 実装方針やアーキテクチャの概要。
4. **受け入れ基準 (Acceptance Criteria)**: どのような状態になれば完了とするか。
5. **テスト戦略**: どのように品質や動作を保証するか。

## プルリクエスト（PR）の送信手順

PRを送信する前に、ローカル環境で以下のチェックを行ってください。

1. **テストの実行**: すべてのユニットテストがパスすることを確認してください。
2. **静的解析 (Linter)**: コードスタイルや静的解析で警告が出ていないことを確認してください。
3. **ビルド検証**: Gemのビルドが正常に行えることを確認してください。
4. **ドキュメント更新**: コードの変更に合わせてドキュメントやREADME等も更新してください。
